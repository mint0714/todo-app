# 1. この章のゴール

この章では、Flaskのコードを詳しく読む前に、ブラウザとサーバがどのように通信しているかを確認します。

第0章では、ToDoアプリをローカルPCで起動し、ブラウザから操作しました。この章では、その画面操作の裏側で発生しているHTTPリクエストとHTTPレスポンスを観察します。

この章を終えると、次のことができるようになります。

- ブラウザとサーバの役割を説明できる
- HTTPリクエストとHTTPレスポンスの流れを説明できる
- GETとPOSTの違いを説明できる
- HTML formがPOSTリクエストを送ることを理解できる
- ToDoアプリの画面操作とURLの関係を説明できる
- ブラウザの開発者ツールで通信を確認できる

# 2. この章の完成イメージ

この章の最後には、ToDoアプリの主な画面操作を、HTTPメソッドとURLに対応づけて説明できる状態を目指します。

```text
ブラウザ
  -> URLへアクセスする
  -> フォームを送信する
  -> リンクやボタンを押す

HTTP通信
  -> GET / でトップページを表示する
  -> POST /add でタスク追加を送信する
  -> GET /categories でカテゴリ管理ページを表示する

Flaskサーバ
  -> リクエストを受け取る
  -> 処理を実行する
  -> HTMLやリダイレクトをレスポンスとして返す
```

スクリーンショット挿入予定:
撮影タイミング: ハンズオン1-2 手順7のあと。
ブラウザの開発者ツールのNetworkタブに、`/add` のPOSTリクエストと、その後の `/` のGETリクエストが表示されている状態。

# 5. 概念説明

## 5-1. この章で出てくる用語

| 用語 | 短い説明 |
| --- | --- |
| URL | ブラウザがアクセスする場所を表す文字列 |
| HTTP | ブラウザとサーバが通信するための約束 |
| リクエスト | ブラウザからサーバへ送る依頼 |
| レスポンス | サーバからブラウザへ返す結果 |
| GET | 主にページやデータを取得するときに使うHTTPメソッド |
| POST | 主にフォーム送信など、データを送るときに使うHTTPメソッド |
| ステータスコード | レスポンスの結果を表す番号 |
| HTML form | 入力内容をサーバへ送信するHTMLの仕組み |
| サーバサイドレンダリング | サーバ側でHTMLを作ってブラウザへ返す方式 |

## 5-2. ブラウザとサーバ

Webアプリは、ブラウザとサーバのやり取りで動きます。

ブラウザは、ユーザーが見る画面を表示します。ユーザーがURLを開いたり、フォームを送信したり、リンクをクリックしたりすると、ブラウザはサーバへリクエストを送ります。

サーバは、ブラウザから届いたリクエストを受け取り、必要な処理を行い、レスポンスを返します。

```text
ブラウザ
  -> リクエスト
  -> サーバ
  -> レスポンス
  -> ブラウザ
```

このToDoアプリでは、Flaskがサーバ側の処理を担当します。

## 5-3. URL

URLは、ブラウザがアクセスする場所を表します。

この教材でよく使うURLは次のとおりです。

```text
http://127.0.0.1:5000
```

| 部分 | 意味 |
| --- | --- |
| `http` | 通信方式 |
| `127.0.0.1` | 自分のPC自身を表すアドレス |
| `5000` | Flaskアプリが待ち受けているポート番号 |
| `/` | トップページを表すパス |

カテゴリ管理ページは、次のURLです。

```text
http://127.0.0.1:5000/categories
```

この場合、パスは `/categories` です。

## 5-4. HTTPリクエストとHTTPレスポンス

HTTPリクエストは、ブラウザからサーバへの依頼です。

例:

```text
GET /
```

これは「トップページをください」という依頼です。

HTTPレスポンスは、サーバからブラウザへの返事です。

例:

```text
200 OK
HTML本文
```

これは「成功しました。画面に表示するHTMLを返します」という返事です。

## 5-5. GETとPOST

GETとPOSTは、HTTPメソッドと呼ばれます。

| HTTPメソッド | 主な用途 | ToDoアプリでの例 |
| --- | --- | --- |
| GET | ページやデータを取得する | トップページを表示する、カテゴリ管理ページを表示する |
| POST | フォームの入力内容を送る | タスクを追加する、カテゴリを追加する、コメントを追加する |

GETは「見る」操作、POSTは「送る」操作と考えると、最初は理解しやすいです。

ただし、実際のWeb開発ではGETとPOST以外にも、PUT、PATCH、DELETEなどのHTTPメソッドがあります。この教材では、まずGETとPOSTを中心に扱います。

## 5-6. ステータスコード

ステータスコードは、サーバが返す結果を番号で表したものです。

| ステータスコード | 意味 | ToDoアプリでの例 |
| --- | --- | --- |
| 200 | 成功 | ページが表示される |
| 302 | 別のURLへ移動する | タスク追加後にトップページへ戻る |
| 404 | 見つからない | 存在しないタスク詳細URLを開く |

ToDoアプリでタスクを追加すると、`POST /add` のあとにトップページへ戻ります。このとき、サーバはリダイレクトという仕組みで、ブラウザに「次は `/` を開いてください」と伝えます。

リダイレクトでは、ステータスコード `302` がよく使われます。

## 5-7. HTML form

HTML formは、入力内容をサーバへ送るためのHTMLの仕組みです。

ToDoアプリのタスク追加フォームは、`templates/index.html` にあります。

```html
<form action='/add' method='POST'>
```

この1行には、重要な情報が2つあります。

| 属性 | 意味 |
| --- | --- |
| `action='/add'` | 送信先のURLは `/add` |
| `method='POST'` | 送信方法はPOST |

つまり、タスク追加フォームで `追加` ボタンを押すと、ブラウザは `POST /add` を送ります。

## 5-8. サーバサイドレンダリング

このToDoアプリは、サーバサイドレンダリングで画面を表示しています。

サーバサイドレンダリングとは、サーバ側でHTMLを作り、完成したHTMLをブラウザへ返す方式です。

このアプリでは、Flaskが `templates/` 配下のHTMLテンプレートを使って画面を作ります。

```text
ブラウザ
  -> GET /
  -> Flask
  -> templates/index.html を使ってHTMLを作る
  -> HTMLをブラウザへ返す
```

## ハンズオン1-4: フォームの`action`と`method`をテンプレートで確認する

目的:
画面上のフォーム操作が、HTMLの `action` と `method` によって決まっていることを確認します。

実行場所:
ローカルPC

変更するファイル:
なし

手順の種類:
確認のみ

作業前の状態:
プロジェクトディレクトリをエディタで開いており、`templates/` 配下のファイルを読める状態です。

手順:

1. `templates/index.html` を開きます。
2. タスク追加フォームを探します。
3. フォームの `action` と `method` を確認します。

確認するHTML:

```html
<form action='/add' method='POST'>
```

4. 完了ボタンのフォームを確認します。

確認するHTML:

```html
<form action='/complete/{{ task.id }}' method='POST' style='display:inline'>
```

5. 削除ボタンのフォームを確認します。

確認するHTML:

```html
<form action='/delete/{{ task.id }}' method='POST' style='display:inline'>
```

6. `templates/categories.html` を開きます。
7. カテゴリ追加フォームの `action` と `method` を確認します。

確認するHTML:

```html
<form action='/categories/add' method='POST'>
```

8. `templates/detail.html` を開きます。
9. コメント追加フォームの `action` と `method` を確認します。

確認するHTML:

```html
<form action='/task/{{ task.id }}/comment' method='POST'>
```

期待される結果:
データを追加、更新、削除するフォームでは、`method='POST'` が使われています。

作業後の状態:
画面操作とHTML formの送信先URLを対応づけられるようになっています。

確認ポイント:

画面上の確認:
フォーム送信ボタンが、タスク追加、完了、削除、カテゴリ追加、コメント追加に対応していることを説明できます。

裏側の確認:
各フォームの `action` が送信先URL、`method` がHTTPメソッドを表していることを説明できます。

> 補足:
> `{{ task.id }}` はJinjaテンプレートの書き方です。
> 実際の画面では、タスクIDに置き換わり、たとえば `/complete/1` のようなURLになります。

# 9. 理解チェックの回答例

1. ブラウザは画面を表示し、ユーザー操作に応じてサーバへリクエストを送ります。サーバはリクエストを受け取り、処理結果をレスポンスとして返します。
2. HTTPリクエストはブラウザからサーバへの依頼です。HTTPレスポンスはサーバからブラウザへの返事です。
3. GETは、ページやデータを取得するときに使います。例として、トップページやカテゴリ管理ページの表示があります。
4. POSTは、フォームの入力内容を送ったり、データを追加、更新、削除したりするときに使います。
5. `action` は、フォームの送信先URLを表します。タスク追加フォームでは `/add` が送信先です。
6. `method` は、フォーム送信に使うHTTPメソッドを表します。タスク追加フォームでは `POST` が使われます。
7. リダイレクトによるものです。サーバが、処理後に別のURLを開くようブラウザへ伝えます。
8. どのURLへ、どのHTTPメソッドで通信したか、ステータスコードが何だったかなどを確認できます。
