# 1. この章のゴール

この章では、自動テストを書く前に「何を、どの種類のテストで確認するか」を整理します。

第0章から第3章では、ToDoアプリを手動で操作し、画面、HTTP、Flask、DBの流れを確認してきました。この章では、その手動確認をテスト観点として整理します。

この章を終えると、次のことができるようになります。

- 手動テストと自動テストの違いを説明できる
- 単体テスト、結合テスト、E2Eテストの違いを説明できる
- E2Eテストに向いている操作を判断できる
- ToDoアプリの重要操作をテスト観点として整理できる
- すべてをE2Eテストにしない理由を説明できる
- テストデータを考える理由を説明できる

# 2. この章の完成イメージ

この章の最後には、ToDoアプリの操作を次のようなテスト観点表として整理できる状態を目指します。

| 操作 | 重要度 | 確認したいこと | 向いているテスト |
| --- | --- | --- | --- |
| タスクを追加する | 高 | 入力したタスクが一覧に表示される | E2E |
| タスクを完了にする | 高 | 完了状態が切り替わる | E2E |
| カテゴリを追加する | 中 | カテゴリが選択肢や絞り込みに表示される | E2E / 結合 |
| コメントを追加する | 中 | タスク詳細にコメントが表示される | E2E |
| 同名カテゴリを追加する | 低 | 重複登録されない | 結合 / 単体 |

テストは、何でも自動化すればよいわけではありません。

重要な操作、壊れやすい処理、手動確認に時間がかかる操作を優先して、自動テストにする範囲を決めます。

# 5. 概念説明

## 5-1. この章で出てくる用語

| 用語 | 短い説明 |
| --- | --- |
| 手動テスト | 人がブラウザや画面を操作して確認するテスト |
| 自動テスト | プログラムで操作や確認を自動実行するテスト |
| 単体テスト | 関数や小さな部品を単独で確認するテスト |
| 結合テスト | 複数の部品を組み合わせて確認するテスト |
| E2Eテスト | ユーザー操作に近い流れをブラウザ越しに確認するテスト |
| テストピラミッド | テストの種類と量のバランスを表す考え方 |
| テスト観点 | 何を確認するかを整理した視点 |
| テストデータ | テスト実行のために用意するカテゴリ、タスク、コメントなどのデータ |

## 5-2. 手動テスト

手動テストは、人がアプリを操作して期待どおりに動くか確認する方法です。

第0章から第3章では、次のような確認を手動で行いました。

- ブラウザでトップページを開く
- タスクを追加する
- タスクを完了にする
- カテゴリを追加する
- タスク詳細を開く
- コメントを追加する

手動テストは、画面の見た目や操作感を確認しやすいです。

一方で、毎回人が同じ操作をする必要があるため、回数が増えると時間がかかります。また、確認漏れも起きやすくなります。

## 5-3. 自動テスト

自動テストは、プログラムで確認作業を実行する方法です。

たとえば、次のような確認を自動化できます。

```text
トップページを開く
  -> タスク入力欄に文字を入れる
  -> 追加ボタンを押す
  -> 入力したタスクが画面に表示されることを確認する
```

自動テストの良い点は、同じ確認を何度も同じ手順で実行できることです。

コードを変更したあとに自動テストを実行すれば、以前できていた操作が壊れていないかを確認しやすくなります。

## 5-4. 単体テスト

単体テストは、関数や小さな部品を単独で確認するテストです。

ToDoアプリで考えるなら、たとえば次のような処理が単体テストに近い対象です。

- 入力値が空かどうかを判定する処理
- カテゴリ名が重複しているかを判定する処理
- 日付や表示用文字列を整形する処理

単体テストは速く実行しやすく、失敗したときに原因を絞り込みやすいです。

ただし、ブラウザ操作全体や、画面、サーバ、DBがつながった流れは単体テストだけでは確認しにくいです。

## 5-5. 結合テスト

結合テストは、複数の部品を組み合わせて確認するテストです。

ToDoアプリでは、たとえば次のような確認が結合テストに向いています。

- View関数がフォーム値を受け取ってDBに保存できる
- カテゴリ追加後、カテゴリ一覧に表示される
- 存在しないタスクIDで詳細画面を開いたときに404になる

結合テストは、関数単体ではなく、アプリ内部の部品同士が正しくつながっているかを確認します。

## 5-6. E2Eテスト

E2Eは、End to Endの略です。

E2Eテストは、ユーザーに近い操作で、アプリ全体の流れを確認するテストです。

ToDoアプリでは、たとえば次のような流れがE2Eテストに向いています。

```text
ブラウザでトップページを開く
  -> タスク名を入力する
  -> 追加ボタンを押す
  -> 追加したタスクが一覧に表示される
```

E2Eテストは、ブラウザ、HTML、Flask、DBがつながった実際の使い方に近い確認ができます。

一方で、単体テストより実行に時間がかかり、失敗したときの原因調査も少し難しくなります。

## 5-7. テストピラミッド

テストピラミッドは、テストの種類と量のバランスを表す考え方です。

よくあるイメージは次のようになります。

```text
        E2Eテスト
      結合テスト
    単体テスト
```

下にある単体テストは、数を多くしやすく、速く実行できます。

上にあるE2Eテストは、ユーザー操作に近い重要な流れを確認できますが、実行時間や保守コストが大きくなりやすいです。

そのため、すべてをE2Eテストにするのではなく、重要な操作を選んでE2Eテストにします。

## 5-8. テスト観点

テスト観点とは、「何を確認したいのか」を整理したものです。

たとえば、タスク追加のテスト観点は次のように分けられます。

| 観点 | 確認内容 |
| --- | --- |
| 画面操作 | タスク名を入力して追加ボタンを押せる |
| 表示 | 追加したタスクが一覧に表示される |
| DB保存 | 再読み込みしてもタスクが残る |
| 関連データ | カテゴリ付きタスクを追加できる |
| 異常系 | 空のタスク名は追加できない |

テスト観点を先に整理すると、あとでテストコードを書くときに迷いにくくなります。

## 5-9. テストデータ

テストデータは、テストを実行するために用意するデータです。

ToDoアプリでは、次のようなデータがテストデータになります。

- カテゴリ
- タスク
- コメント
- 完了済みのタスク
- 未完了のタスク
- カテゴリなしのタスク

自動テストでは、毎回同じ状態から始めることが重要です。

たとえば、前回のテストで作ったタスクが残っていると、次のテスト結果が変わってしまいます。

第5章では、テストを安定して実行するために、開発用DBとテスト用DBを分ける考え方を扱います。

## ハンズオン4-1: 手動確認している操作を書き出す

目的:
これまで手動で確認してきたToDoアプリの操作を一覧化します。

実行場所:
ローカルPC

変更するファイル:
なし。任意で `docs/test_plan.md` に学習メモを作成してもよい

手順の種類:
実行する手順

作業前の状態:
ToDoアプリをブラウザで操作でき、これまでの章で主要な画面操作を確認しています。

手順:

1. ToDoアプリをブラウザで開きます。
2. 画面上部のナビゲーションと各画面のフォームを確認します。
3. 手動確認している操作を一覧にします。

例:

| 操作 | 画面 | 関係するURL |
| --- | --- | --- |
| トップページを表示する | タスク一覧 | `/` |

| カテゴリを追加する | カテゴリ管理 | `/categories/add` |
| タスク詳細を表示する | タスク詳細 | `/task/<task_id>` |
| コメントを追加する | タスク詳細 | `/task/<task_id>/comment` |

4. 追加で確認したい操作があれば、表に加えます。

期待される結果:
ToDoアプリで手動確認している操作を一覧として説明できます。

作業後の状態:
テスト対象になり得る操作の候補が整理されています。

確認ポイント:

画面上の確認:
タスク一覧、カテゴリ管理、タスク詳細の各画面で、どの操作ができるかを確認できます。

裏側の確認:
各操作がどのURLやView関数につながるかを、第1章、第2章の内容と結びつけて説明できます。

## ハンズオン4-2: 重要操作に優先順位を付ける

目的:
すべてを同じ重さでテストするのではなく、重要な操作から優先して整理します。

実行場所:
ローカルPC

変更するファイル:
なし。任意で `docs/test_plan.md` に学習メモを作成してもよい

手順の種類:
実行する手順

作業前の状態:
ハンズオン4-1で、ToDoアプリの主な操作を一覧化しています。

手順:

1. ハンズオン4-1の操作一覧を見ます。
2. 各操作に重要度を付けます。
3. 重要度は、まず `高`、`中`、`低` の3段階で考えます。

重要度の目安:

| 重要度 | 判断の目安 |
| --- | --- |
| 高 | アプリの中心機能。壊れると主要な使い方ができない |
| 中 | よく使うが、中心機能の補助にあたる |
| 低 | 頻度が低い、または他のテストである程度確認できる |

例:

| 操作 | 重要度 | 理由 |
| --- | --- | --- |
| タスクを追加する | 高 | ToDoアプリの中心機能だから |
| タスクを完了にする | 高 | タスク管理の主要操作だから |
| コメントを追加する | 中 | 詳細画面の重要な補助機能だから |
| 同名カテゴリを追加する | 低 | 重要だが、画面全体より内部処理寄りだから |

期待される結果:
ToDoアプリの操作を、重要度つきで整理できます。

作業後の状態:
自動テストにする候補を、優先順位で選びやすくなっています。

確認ポイント:

画面上の確認:
画面操作の中で、ユーザーにとって重要な流れを説明できます。

裏側の確認:
重要度を、ユーザー影響や壊れたときの困り方で説明できます。

> 補足:
> 最初から完璧な優先順位にする必要はありません。
> 自動テストを書き始める前に、どの操作から守りたいかを考えることが大切です。

## ハンズオン4-3: E2Eで確認する操作を選ぶ

目的:
E2Eテストに向いている操作と、別の種類のテストに向いている操作を分けます。

実行場所:
ローカルPC

変更するファイル:
なし。任意で `docs/test_plan.md` に学習メモを作成してもよい

手順の種類:
実行する手順

作業前の状態:
ToDoアプリの操作一覧と重要度を整理しています。

手順:

1. 重要度が `高` の操作を確認します。
2. その操作が、ブラウザ、Flask、DBをまたぐ流れか確認します。
3. ユーザーに近い操作で確認したいものをE2E候補にします。
4. 内部ロジックだけで確認しやすいものは、単体テストや結合テスト候補にします。

E2Eに向いている操作の目安:

| 判断項目 | E2Eに向いている理由 |
| --- | --- |
| ブラウザ操作が重要 | 入力、クリック、画面表示まで確認したい |
| 複数の機能がつながる | 画面、View関数、DB保存をまとめて確認したい |
| ユーザー影響が大きい | 壊れると主要な使い方ができない |
| 手動確認が面倒 | 毎回同じ操作を人が行う負担が大きい |

例:

| 操作 | 向いているテスト | 理由 |
| --- | --- | --- |
| タスクを追加する | E2E | 入力、POST、DB保存、一覧表示まで確認したい |
| タスクを完了にする | E2E | ボタン操作と表示切り替えを確認したい |
| 同名カテゴリを追加する | 結合 | DB保存前の条件分岐を確認したい |
| 存在しないタスクIDを開く | 結合 | 404の処理を確認したい |

期待される結果:
E2Eテストにする操作と、単体テストや結合テストに向いている操作を分けられます。

作業後の状態:
次章以降でE2Eテスト化する操作の候補が見えています。

確認ポイント:

画面上の確認:
ブラウザ操作として確認したい重要な流れを選べます。

裏側の確認:
すべてをE2Eにしない理由を、実行時間や保守コストの観点で説明できます。

## ハンズオン4-5: テストデータの扱い方を整理する

実行場所:
ローカルPC

変更するファイル:
なし。任意で `docs/test_plan.md` に学習メモを作成してもよい

手順の種類:
実行する手順

# 8. 理解チェック

次の質問に、自分の言葉で答えてみましょう。

1. 手動テストと自動テストの違いは何ですか？
2. 単体テストは、どのような確認に向いていますか？
3. 結合テストは、どのような確認に向いていますか？
4. E2Eテストは、どのような確認に向いていますか？
5. すべての確認をE2Eテストにしない方がよい理由は何ですか？
6. ToDoアプリでE2Eテストに向いている操作を2つ挙げてください。
7. テスト観点表を作る目的は何ですか？
8. 自動テストでテストデータをそろえる必要があるのはなぜですか？

# 9. 理解チェックの回答例

1. 手動テストは人が画面を操作して確認する方法です。自動テストはプログラムで確認作業を実行する方法です。
2. 単体テストは、関数や小さな部品の動きを確認するのに向いています。速く実行でき、失敗原因を絞り込みやすいです。
3. 結合テストは、View関数とDBなど、複数の部品が正しくつながっているかを確認するのに向いています。
4. E2Eテストは、ブラウザ操作から画面表示、サーバ処理、DB保存まで、ユーザーに近い流れを確認するのに向いています。
5. E2Eテストは実行時間や保守コストが大きくなりやすく、失敗したときの原因調査も難しくなりやすいためです。
6. 例として、タスク追加、タスク完了、コメント追加があります。どれもユーザー操作、サーバ処理、DB保存、画面表示がつながるためです。
7. 何を確認するか、どの種類のテストで確認するか、どのデータが必要かを整理し、テストコードを書く前に迷いを減らすためです。
8. 前回のテストで作ったデータが残ると、次回のテスト結果が変わることがあるためです。毎回同じ状態から始めることで、安定したテストになります。
