この章を終えると、次のことができるようになります。

- 本番DBとテストDBを分ける理由を説明できる

## 5-2. なぜDBを分けるのか

自動テストでは、毎回同じ状態から始めることが重要です。

もし開発用DBとテスト用DBが同じだと、次のような問題が起きます。

- 手動確認で作ったタスクがテスト結果に影響する
- テスト実行で開発用データが増える
- 削除テストで手動確認用のデータが消える
- 自分のPCでは成功するが、別のPCやCIでは失敗する

そのため、DBは目的ごとに分けて考えます。

| DBの種類 | 使う場面 | 特徴 |
| --- | --- | --- |
| 開発用DB | ローカルで手動確認するとき | 学習中のデータが残ってもよい |
| テスト用DB | pytestやPlaywrightを実行するとき | 毎回同じ状態に作り直せる |
| 本番用DB | Render上で公開アプリを動かすとき | 実ユーザーのデータを扱う |

> 注意:
> テストで本番用DBを使ってはいけません。
> 自動テストはデータを追加、更新、削除するため、本番データを壊す危険があります。

## 5-3. 現在のDB設定

現在の `app.py` には、次のDB設定があります。

```python
app.config['SQLALCHEMY_DATABASE_URI'] = 'sqlite:///todo.db'
```

これは、SQLiteの `todo.db` を使う設定です。

ローカル開発では分かりやすい構成ですが、テストや本番を考えると課題があります。

| 課題 | 理由 |
| --- | --- |
| DB接続先がコードに直接書かれている | 環境ごとに切り替えにくい |
| テスト用DBを指定しにくい | 自動テストで毎回同じ状態を作りにくい |
| 本番DBに切り替えにくい | RenderではPostgreSQLなどを使う想定がある |

この章では、この課題をどう改善するかを設計として整理します。

## 5-4. 環境変数

環境変数は、アプリの外側から渡す設定値です。

コードの中に固定値を書く代わりに、実行環境ごとに設定を変えられます。

例:

```text
ローカル開発
  DATABASE_URL=sqlite:///todo.db

自動テスト
  DATABASE_URL=sqlite:///test_todo.db

本番
  DATABASE_URL=postgresql://...
```

環境変数を使うと、同じアプリコードでも、実行する場所によってDB接続先を切り替えられます。

## 5-5. `DATABASE_URL`

`DATABASE_URL` は、DB接続先を表す環境変数名としてよく使われます。

ToDoアプリでも、将来的には次のような方針にできます。

```text
DATABASE_URL が設定されている
  -> DATABASE_URL のDBを使う

DATABASE_URL が設定されていない
  -> ローカル開発用SQLiteを使う
```

この方針にすると、ローカルでは簡単に起動でき、本番やテストでは外からDB接続先を指定できます。

## 5-6. テスト用SQLite

E2Eテストでは、まずテスト用SQLiteを使う方針にすると学びやすいです。

理由は、SQLiteが軽量で、ローカルPCでもCIでも扱いやすいためです。

テスト用DBの例:

```text
sqlite:///test_todo.db
```

ただし、テスト用DBでは、毎回同じ状態を作る必要があります。

たとえば、テスト開始時に次のような処理を行います。

```text
既存のテスト用DBを初期化する
  -> テーブルを作成する
  -> 必要なテストデータを入れる
  -> テストを実行する
  -> 必要に応じて後片付けする
```

## 5-7. アプリケーションファクトリ

アプリケーションファクトリは、Flaskアプリを関数で作る構成です。

よく使われる関数名は `create_app()` です。

イメージ:

```python
def create_app(config=None):
    app = Flask(__name__)
    app.config.from_mapping(config or {})
    db.init_app(app)
    return app
```

現在の `app.py` は、ファイルを読み込んだ時点で `app = Flask(__name__)` が実行されます。

これは小さなアプリでは分かりやすい構成です。

一方で、テストや本番設定を切り替えたい場合は、`create_app()` のような形にすると、テスト用設定を渡してアプリを作りやすくなります。

| 構成 | 特徴 |
| --- | --- |
| 直接 `app = Flask(__name__)` | 小さなアプリでは分かりやすい |
| `create_app()` 形式 | 環境ごとに設定を切り替えやすい |

## 5-8. DB初期化

現在の `app.py` には、次のコードがあります。

```python
with app.app_context():
    db.create_all()
```

これは、アプリ起動時にDBテーブルを作成するための処理です。

学習用アプリでは便利ですが、テストや本番を考えると、初期化のタイミングを分けて考える必要があります。

| 場面 | DB初期化の考え方 |
| --- | --- |
| 開発 | 起動時にテーブルを作ってもよい |
| テスト | テスト開始時に毎回作り直したい |
| 本番 | 不用意に初期化や削除を行わない |

テストでは、毎回同じ状態から始めるために、テスト用DBを作り直す方針が重要になります。

## 5-9. `debug=True`

現在の `app.py` では、最後に次のように起動しています。

```python
app.run(debug=True)
```

`debug=True` は、開発中には便利です。

たとえば、エラー内容が詳しく表示されたり、コード変更時にアプリが自動で再読み込みされたりします。

しかし、本番環境では使いません。

理由は、エラーの詳細や内部情報が外部に見えてしまう危険があるためです。

| 環境 | `debug=True` |
| --- | --- |
| ローカル開発 | 使ってよい |
| 自動テスト | 必須ではない |
| 本番 | 使わない |

def create_app(config=None):

```python

})
```

| 設定切り替え | やや難しい | 設定を渡しやすい |
| テスト | テスト用設定を差し込みにくい | テスト用アプリを作りやすい |
| CI/CD | 環境ごとの設定管理がしにくい | 環境変数と組み合わせやすい |

# 8. 理解チェック

次の質問に、自分の言葉で答えてみましょう。

1. 開発用DB、テスト用DB、本番用DBは、それぞれ何のために使いますか？
2. テストで開発用DBや本番用DBを使うと、どのような問題が起きますか？
3. 環境変数を使うと、どのような利点がありますか？
4. `DATABASE_URL` は何を表すために使いますか？
5. テスト用SQLiteを使う利点は何ですか？
6. `create_app()` 形式にすると、なぜテストしやすくなりますか？
7. テスト用DBを毎回初期化する理由は何ですか？
8. `debug=True` を本番で使わない方がよい理由は何ですか？

# 9. 理解チェックの回答例

1. 開発用DBはローカルで手動確認するときに使います。テスト用DBは自動テストで使います。本番用DBは公開アプリが実ユーザーのデータを保存するために使います。
2. テストデータが開発用DBに残ったり、本番データを壊したりする危険があります。また、DBの状態が人や環境によって違うと、テスト結果が安定しません。
3. コードを書き換えずに、実行環境ごとに設定を変えられます。ローカル、テスト、CI、本番でDB接続先を切り替えやすくなります。
4. DB接続先を表すために使います。SQLiteやPostgreSQLなど、使うDBの場所や接続情報を指定します。
5. 軽量で扱いやすく、ローカルPCやCIで準備しやすいことです。学習段階のE2Eテストにも向いています。
6. テスト用の設定を渡してFlaskアプリを作れるためです。DB接続先やテストモードを切り替えやすくなります。
7. 前回のテストで作られたデータが残ると、次回のテスト結果が変わることがあるためです。毎回同じ状態から始めることで、テストが安定します。
8. エラー詳細や内部情報が外部に見える危険があるためです。本番では安全性を優先し、デバッグモードは無効にします。
