# Flask ToDoアプリ教材 内容計画書

この文書は、Flask製ToDoアプリを題材にした教材シリーズで「何を、どの順番で、どの到達点まで扱うか」を定義する計画書です。

各章のMarkdown本文を作成するときは、この計画書で章の内容を確認し、`docs/material_reference.md` で書き方の基準を確認します。

```text
docs/curriculum_plan.md
  -> 教材内容の計画。何を教えるか、各章で何を作るかを定義する。

docs/material_reference.md
  -> 教材作成の基準。章構成、表記、ハンズオン形式などを定義する。

docs/chapterXX_xxx.md
  -> 各章本文の正本。確認後にDOCXへ変換する。
```

## 1. 教材全体の目的

この教材の目的は、Webアプリケーション初学者が、1つのFlaskアプリを題材にして次の流れを自分で実装できるようになることです。

- Webアプリケーションの基本的な仕組みを理解する
- Flaskアプリの構造を読めるようになる
- データベースと画面操作のつながりを理解する
- Python版PlaywrightでE2Eテストを書けるようになる
- GitHub ActionsでCIを構築できるようになる
- RenderでデプロイとCDを設定できるようになる
- 変更、テスト、PR、デプロイの一連の流れを説明できるようになる

## 2. 想定読者

- Webアプリケーション開発の初学者
- Pythonの基本文法は少し触ったことがある
- Flask、テスト自動化、CI/CDはこれから学ぶ
- GitやGitHubの操作にまだ慣れていない
- 手順を追いながら、自分の手で実装して理解したい

## 3. 教材で扱う技術

| 分野 | 扱う技術 |
| --- | --- |
| Webアプリ | Flask、Jinja、HTML form、CSS |
| データベース | SQLite、SQLAlchemy、PostgreSQLの入口 |
| テスト | pytest、Python版Playwright |
| CI | GitHub Actions |
| CD | Render、Render Auto Deploy |
| バージョン管理 | Git、GitHub、branch、pull request |

## 4. 教材で扱わないこと

本教材は、Webアプリの基本、テスト自動化、CI/CDの入口に集中します。以下は主題にしません。

- フロントエンドフレームワークの本格導入
- JavaScript中心のSPA開発
- 高度なCSSデザイン
- 認証、認可、ユーザー管理
- Dockerの本格運用
- Kubernetesなどの大規模インフラ
- 本格的な監視基盤やSRE領域
- 大規模チーム開発の詳細な運用ルール

必要な場合は補足として触れますが、各章の主目的にはしません。

## 5. 教材全体の学習ステージ

教材全体を3つのステージに分けます。

| ステージ | 対象章 | 目的 |
| --- | --- | --- |
| Stage 1: Webアプリの土台 | 第0章〜第3章 | 開発環境、HTTP、Flask、DBの基本を理解する |
| Stage 2: テスト自動化 | 第4章〜第8章 | テスト設計、Playwright導入、E2Eテスト、失敗時の調査を学ぶ |
| Stage 3: CI/CDと運用 | 第9章〜第13章 | GitHub Actions、Render、CD、実務的な運用を学ぶ |

## 6. 教材全体の完成イメージ

最終的には、次の流れを自分で作れる状態を目指します。

```text
ローカルPC
  -> Flaskアプリを開発する
  -> PlaywrightでE2Eテストを実行する
  -> 作業ブランチをGitHubへpushする

GitHub
  -> Pull Requestを作成する
  -> GitHub Actionsで自動テストを実行する
  -> テストが成功したらmainへマージする

Render
  -> mainへの変更を検知する
  -> CI成功後に自動デプロイする
  -> 本番URLでアプリを確認する
```

## 7. 教材全体の作業前提

各章のMarkdown本文を作成するときは、次の前提をそろえます。ここをそろえることで、章をまたいだ説明のズレを減らします。

### Git運用前提

本教材では、基本ブランチを `main` として説明します。

ただし、学習用リポジトリやチームの運用で `develop` を使っている場合は、教材中の `main` を `develop` に読み替えます。

```text
教材上の基本形:
作業ブランチ
  -> pull request
  -> main
  -> CI成功
  -> Renderへデプロイ

develop運用の場合:
作業ブランチ
  -> pull request
  -> develop
  -> CI成功
  -> 必要に応じて main へ反映
  -> Renderへデプロイ
```

各章の作業ブランチ名は、原則として次の形式にします。

```text
chapterXX-practice
```

例:

```text
chapter00-practice
chapter07-practice
chapter09-practice
```

### 章本文の整合ルール

各章のMarkdown本文では、読者が「いま何をしていて、何が変わったのか」を追えるようにします。

- 章ごとに「作業前の状態」と「作業後の状態」を必要に応じて書く
- 各ハンズオンに「変更するファイル」を書く
- ファイルを編集しない場合も `なし` と明記する
- 実際に行う手順、確認のみの手順、任意の手順を区別する
- 確認ポイントは、必要に応じて「画面上の確認」と「裏側の確認」に分ける
- OS差があるコマンドは、Linux / macOS と Windows PowerShell を分けて書く
- スクリーンショット枠には、何の画面かに加えて撮影タイミングを書く

### 用語の初出ルール

専門用語は、初めて出る章で短く説明します。

各章では、必要に応じて概念説明の冒頭や表の中で、初出用語を整理します。

例:

| 章 | 初出用語の例 |
| --- | --- |
| 第1章 | HTTP、リクエスト、レスポンス、GET、POST、ステータスコード |
| 第2章 | ルーティング、View関数、テンプレート、リダイレクト |
| 第3章 | Model、Table、Record、Primary Key、Foreign Key |
| 第6章 | Playwright、ブラウザ自動操作、fixture |
| 第9章 | workflow、job、step、runner |
