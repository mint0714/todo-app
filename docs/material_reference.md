# Flask ToDoアプリ教材 DOCX作成リファレンス

Webアプリ開発・テスト自動化・CI/CD教材シリーズの共通ガイドです。

このリファレンスは、各章のDOCX教材を同じ品質・同じ読みやすさで作るための正本として扱います。第0章だけでなく、Flask、DB、Playwright、GitHub Actions、Renderの各章にも共通で適用します。

## 1. 基本方針

- 教材は「説明資料」ではなく「手を動かして完了できる手順書」として作る。
- 読者はWebアプリ初学者である前提にする。
- 専門用語は最初に短く説明する。
- 各章の構成、ラベル、表記、確認方法を統一する。
- 作業場所を明示する。
  - 例: ローカルPC、GitHub画面、GitHub Actionsログ、Render画面
- 各ハンズオンには目的、手順、期待結果、確認ポイントを入れる。
- 各章や各ハンズオンでは、必要に応じて「作業前の状態」と「作業後の状態」を明示する。
- 実際に手を動かす手順と、画面や流れを確認するだけの手順を区別する。
- 章末にエラー集の専用セクションは入れない。
- 必要な注意は、各手順内の注意ボックスで短く扱う。

## 3. 章共通テンプレート

各章は原則として次の順序で作成します。章によって不要な項目がある場合も、できるだけ順序は崩さないようにします。

1. 表紙
2. 目次
3. この章のゴール
4. この章の完成イメージ
5. この章で使うファイル・コマンド
6. 前提確認
7. 概念説明
8. ハンズオン
9. 動作確認
10. 理解チェック
11. 理解チェックの回答例
12. 次章への接続

### 除外ルール

章末のエラー集セクションは作りません。

トラブル時の補足は、該当手順の近くに注意ボックスまたは補足ボックスとして短く入れます。

## 4. 表紙の標準

表紙には、読者が最初に知りたい情報だけを置きます。文章量を増やしすぎないようにします。

| 項目 | 書き方 |
| --- | --- |
| 教材タイトル | Flask ToDoアプリで学ぶ Webアプリ開発・テスト自動化・CI/CD |
| 章タイトル | 第N章 章名 |
| 対象 | Webアプリケーション初学者 |
| 所要時間 | 例: 60〜90分 |
| 難易度 | 入門 / 基礎 / 実践 |
| この章で作るもの | 例: ローカル起動環境、E2Eテスト、CI workflow |
| この章で変更するファイル | 例: なし、`app.py`、`tests/e2e/test_todo_flow.py` |

## 5. 目次の扱い

- DOCXでは見出しスタイルを必ず使い、WordやLibreOfficeで目次を更新できる状態にする。
- 自動目次を完全生成できない場合は、「目次」ページに主要見出しを手動で列挙する。
- 目次には第1階層と第2階層までを載せる。
- 細かすぎる手順番号は目次に載せない。

## 6. 見出しルール

| レベル | 用途 | 例 |
| --- | --- | --- |
| Heading 1 | 章内の大きな区切り | `1. この章のゴール` |
| Heading 2 | 概念やハンズオンのまとまり | `7-1. トップページを表示するテストを書く` |
| Heading 3 | 補助的な小見出し | `期待される結果` |

- 見出し番号は章番号とは別に、章内で読みやすい番号を付ける。
- 見出しは長くしすぎない。
- 1行で意味が伝わる長さにする。
- 章をまたいで同じ意味の見出し名を使う。
  - 例: この章のゴール、動作確認、理解チェック

## 7. ラベルとボックスの標準

教材内の情報種別を、見た目と言葉で統一します。

| ラベル | 用途 | 色の目安 |
| --- | --- | --- |
| 目的 | その手順で何を達成するか | 薄い青 |
| 補足 | 知らなくても進めるが理解が深まる説明 | 薄い黄 |
| 注意 | 失敗や危険を避けるための短い警告 | 薄い赤 |
| 確認 | 手順後に見るべき結果 | 薄い緑 |
| メモ | 学習者が書き込む余白や問い | 薄いグレー |

### 注意ボックス

注意ボックスは短くします。長い説明を入れると読まれにくくなります。

詳しい説明が必要な場合は、注意ボックスではなく本文に分けます。

### 確認ボックス

確認ボックスには「何が見えれば成功か」を具体的に書きます。

例:

```text
確認:
ブラウザでToDoリスト画面が表示される。
```

## 8. ハンズオンの標準構成

ハンズオンは1つの作業単位を小さく区切ります。1つのハンズオンで複数の目的を混ぜないようにします。

```text
ハンズオンN-M: タイトル

目的:
この手順で達成することを書く。

実行場所:
ローカルPC / GitHub画面 / GitHub Actionsログ / Render画面

変更するファイル:
なし / app.py / templates/index.html / tests/e2e/test_todo_flow.py など

作業前の状態:
この手順を始める前に、画面、ファイル、DB、Git状態がどうなっているかを書く。

手順:
1. 具体的な操作を書く
2. 実行するコマンドを書く
3. 画面で確認する内容を書く

期待される結果:
成功時に表示される画面、ログ、ファイル、テスト結果を書く。

作業後の状態:
この手順を終えたあとに、画面、ファイル、DB、Git状態がどう変わったかを書く。

確認ポイント:
読者が自分で成功判定できる内容を書く。
```

### ハンズオンの補足ルール

- 「変更するファイル」は、ファイルを編集しない場合も `なし` と明記する。
- 手順の性質が分かりにくい場合は、次のようにラベルを付ける。
  - 実行する手順: 読者が実際にコマンドや画面操作を行う。
  - 確認のみ: 画面や設定、ログを見るだけで、変更は行わない。
  - 任意: 学習環境や講師の指示により実行する。
- 初学者が「何が変わったのか」を追えるように、作業前と作業後の状態を短く書く。
- 確認ポイントは、必要に応じて「画面上の確認」と「裏側の確認」に分ける。
  - 画面上の確認: ブラウザ表示、ボタン、フォーム、GitHub画面など。
  - 裏側の確認: ファイル、DB、ログ、Git状態、テスト結果など。
- 1つのハンズオンで、実装、確認、Git操作を詰め込みすぎない。

## 9. コマンド表記ルール

- コマンドは本文に埋め込まず、必ずコードブロックとして分ける。
- 実行場所を直前に明示する。
  - 例: 実行場所: ローカルPCのターミナル
- プロンプト記号の `$` は原則付けない。
  - コピー時に混乱するため。
- 環境差があるコマンドは表で分ける。
  - 例: Linux/macOS と Windows PowerShell
- Pythonコマンドの環境差は、章内で最初に明記する。
  - 例: Linux / macOS は `python3`、Windows PowerShell は `python`
- 同じ章の中で、同じ環境に対するコマンド表記を不用意に混在させない。
- 破壊的なコマンドは教材内で扱わない。
- 必要な場合は注意ボックスを付ける。

| 環境 | 仮想環境の有効化コマンド |
| --- | --- |
| Linux / macOS | `source venv/bin/activate` |
| Windows PowerShell | `.\\venv\\Scripts\\Activate.ps1` |

## 10. ファイル名・URL・用語の表記

| 対象 | 表記ルール | 例 |
| --- | --- | --- |
| ファイル名 | 等幅表記にする | `app.py` |
| ディレクトリ | 末尾に `/` を付ける | `templates/` |
| URL | 改行で独立させる | `http://127.0.0.1:5000` |
| コマンド | コードブロックにする | `python app.py` |
| GitHub Actions | 正式名称で書く | GitHub Actions |
| Playwright | Python版であることを必要に応じて明記する | Python版Playwright |

## 11. スクリーンショット枠

人間が読む教材では、完成画面のイメージが重要です。

実際の画像を入れない段階でも、スクリーンショット差し込み枠を置きます。

```text
スクリーンショット挿入予定:
撮影タイミング: ハンズオンN-M 手順Xのあと。
ToDoリスト画面がブラウザに表示されている状態。
URLバーには http://127.0.0.1:5000 が表示されている。
```

- スクリーンショット枠には「何の画面か」「どの状態か」を書く。
- スクリーンショット枠には、可能な限り「撮影タイミング」を書く。
- 画面が変わる重要ポイントにだけ入れる。
- 入れすぎると資料が重くなる。
- CIやRenderの章では、設定画面や実行ログのスクリーンショット枠を入れる。

## 12. 動作確認の標準

ハンズオンの最後には、読者が成功を判断できる動作確認を入れます。

画面操作だけでなく、ファイル、DB、ログ、Git状態、テスト結果などが関係する章では、確認を分けて書きます。

```text
画面上の確認:
ブラウザで追加したタスクが表示される。

裏側の確認:
git status で想定したファイルだけが変更されている。
pytest の結果が pass になっている。
```

| 章の種類 | 確認方法の例 |
| --- | --- |
| Flask基礎 | ブラウザで画面が表示される、フォーム送信後に一覧へ戻る |
| DB基礎 | データが保存され、再読み込み後も表示される |
| Playwright | `pytest` でテストがpassする |
| GitHub Actions | PR上のチェックが成功する |
| Render | 本番URLでアプリが表示される |
| CD | mainマージ後にRenderのデプロイが走る |

## 13. 理解チェックと回答例

- 各章末に理解チェックを入れる。
- 質問は5〜8問程度にする。
- 多すぎると負担になる。
- 回答例を必ず付ける。
- 独学でも確認できるようにする。
- 回答例は丸暗記ではなく、考え方が分かる短い文章にする。

例:

```text
理解チェック:
1. なぜ本番DBとテストDBを分ける必要がありますか？
2. E2Eテストで確認すべき操作はどのようなものですか？

回答例:
1. テスト実行で本番データを壊さないため。また、毎回同じ状態でテストを始めるため。
2. ユーザーにとって重要で、複数の機能がつながる操作。例: タスク追加、詳細表示、コメント追加。
```

## 14. 章末の締め方

各章末は、学習者が達成感を持てるように短くまとめます。

```text
この章でできるようになったこと:
- Flaskアプリをローカルで起動できる
- Gitで作業ブランチを作成できる
- PRが何のためにあるか説明できる

次章への接続:
次章では、ブラウザとサーバがどのように通信しているかを見ます。
今回操作したタスク追加やカテゴリ追加の裏側で、HTTPリクエストがどのように送られているかを確認します。
```

## 15. 見た目の標準

- 本文フォントは `Yu Gothic` 10.5ptを基本にする。
- 見出し1は16pt、見出し2は13pt、見出し3は11ptを目安にする。
- コマンドは `Consolas` 9.5pt、薄いグレー背景にする。
- 本文の1段落は長くしすぎない。
- 3〜4行を超える場合は分ける。
- 表は説明を圧縮できる場合だけ使う。
- 表だらけにしない。
- ページごとに余白を残す。
- 情報を詰め込みすぎない。

## 19. 運用ルール

- このリファレンスを先に更新してから、各章DOCXを作成または改訂する。
- 章ごとに独自フォーマットを増やさない。
- 必要な変更は全章共通ルールとして検討する。
- 初学者が迷う箇所は、説明を増やすよりも「目的」「実行場所」「期待結果」を明確にする。
- アプリ本体の実装と教材作成は分けて管理する。
- 教材作成だけの依頼ではアプリ本体を編集しない。
