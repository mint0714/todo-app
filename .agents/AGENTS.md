# Codex 日本語応答ルール

Codexは、このリポジトリでの回答、計画、実装報告、レビュー結果を日本語で行うこと。単に日本語に翻訳するのではなく、読み手が作業内容、判断理由、確認方法を追える技術報告として書くこと。

## 基本方針

回答の最初に、結論を1〜2文で述べること。結論では、何をするのか、何をしたのか、または何が問題なのかを明確にすること。

説明では、「これ」「それ」「上記」「該当箇所」「修正しました」「対応しました」だけで話を進めないこと。対象のファイル名、関数名、クラス名、画面名、コマンド名、設定キーを具体的に書くこと。

短文を連続で並べるだけの報告を避けること。各段落は、「何をしたか」「なぜ必要か」「どう確認するか」が分かる文章にすること。

## 実装前の説明

実装に入る前に、作業対象、変更方針、影響範囲、確認方法を説明すること。

作業対象では、変更予定のファイル名、関数名、画面名、設定ファイル名を明示すること。対象がまだ特定できていない場合は、調査対象として見る予定の場所を説明すること。

変更方針では、なぜその変更が必要なのかを説明すること。単に「改善します」「修正します」と書かず、現在の問題と対応方針のつながりを説明すること。

確認方法では、実行する予定のコマンド、目視確認する画面、確認できる成功条件を説明すること。

## 実装後の報告

実装後の報告では、変更したファイル、変更内容、理由、確認結果、未確認の点を分けて説明すること。

複数ファイルを変更した場合は、次の表で整理すること。

| ファイル | 変更内容 | 理由 |
|---|---|---|

確認結果では、実行したコマンドを具体的に書くこと。例として、`pytest`、`npm test`、`npm run lint`、`npm run build`などを実行した場合は、その結果を説明すること。

確認できていないことがある場合は、「未確認」と明示すること。確認していないのに「問題ありません」と書かないこと。

## 日本語の書き方

主語を省略しすぎないこと。特に、処理、ファイル、関数、テスト、ユーザー操作のどれについて説明しているのかが曖昧にならないようにすること。

専門用語を使う場合は、初出で短く補足すること。初心者が読む可能性があるため、略語やツール名だけで説明を終えないこと。

一文が長くなりすぎる場合は、文章を分けること。ただし、短文を箇条書きのように連続させず、文と文の関係が分かるようにつなげること。

## 禁止する報告

「修正しました」だけで終えないこと。

「確認しました」だけで終えないこと。

「テストしました」だけで終えないこと。

「問題ありません」だけで終えないこと。

必ず、何を、なぜ、どのように確認したのかを説明すること。
